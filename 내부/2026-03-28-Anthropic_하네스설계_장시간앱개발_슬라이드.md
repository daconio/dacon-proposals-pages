# 장시간 실행 앱 개발을 위한 하네스 설계

> 원문: Anthropic Engineering Blog — *Harness Design for Long-Running Application Development*
> 저자: Prithvi Rajasekaran (Anthropic Labs) | 2026.03.24

---

## 핵심 질문

고품질 프론트엔드 디자인과 완전 자율 앱 개발, 두 마리 토끼를 어떻게 잡을 것인가?

- GAN(Generative Adversarial Network)에서 영감을 받은 **Generator-Evaluator 멀티 에이전트 시스템**
- 주관적 판단("이 디자인 좋은가?")을 **구체적이고 채점 가능한 기준**으로 변환

---

## 01. 핵심 문제점

### 왜 단순한 구현은 실패하는가

**1. 컨텍스트 윈도우 열화 (Context Window Degradation)**

- 긴 작업에서 컨텍스트가 채워지면 모델 일관성 저하
- "컨텍스트 불안": 한계에 가까워지면 조기에 작업을 마무리하려는 경향
- 해결: 컨텍스트 압축보다 **완전 리셋**이 더 효과적

**2. 자기평가 편향 (Self-Evaluation Bias)**

- 에이전트가 자기 결과물을 자신 있게 칭찬하는 현상 (품질이 평범해도)
- 디자인 같은 주관적 태스크에서 특히 치명적
- 해결: **생성 에이전트와 평가 에이전트를 분리**

---

## 02. 프론트엔드 디자인 품질 평가

### 4가지 평가 기준

| 기준 | 핵심 질문 |
|------|----------|
| **디자인 품질** | 부분의 집합이 아니라 하나의 일관된 전체로 느껴지는가? |
| **독창성** | 템플릿 기본값이나 AI 패턴(보라색 그라데이션 + 흰 카드)이 아닌 의도적 선택의 증거가 있는가? |
| **완성도** | 타이포그래피 위계, 간격 일관성, 색상 조화, 대비율의 기술적 실행 수준은? |
| **기능성** | 사용자가 인터페이스를 이해하고, 주요 액션을 찾고, 직관적으로 완료할 수 있는가? |

### Generator-Evaluator 시스템

- **Generator**: Claude Agent SDK 기반, HTML/CSS/JS 프론트엔드 생성
- **Evaluator**: Playwright MCP로 실제 페이지와 상호작용 → 스크린샷 촬영 → 기준별 채점
- **반복 횟수**: 생성당 5~15회 반복, 전체 실행 시간 최대 4시간
- Generator가 평가 점수에 따라 **현재 방향 개선** vs **완전히 새로운 접근** 전략적 선택

### 놀라운 결과: 네덜란드 미술관 사례

- 반복 9회까지: 세련되었지만 **전형적인** 디자인
- **반복 10회**: 전체 컨셉을 재구상 → CSS 원근법으로 렌더링된 3D 방, 체커보드 바닥, 벽에 자유롭게 배치된 작품 — **공간적 경험**으로 탈바꿈
- 프롬프트 문구 자체가 모델을 제네릭 디자인에서 이탈하도록 유도

## 03. 풀스택 코딩으로 확장

### 3-에이전트 아키텍처

| 에이전트 | 역할 | 핵심 포인트 |
|---------|------|-----------|
| **Planner** | 1~4문장 프롬프트 → 종합 제품 명세서 확장 | 세부 구현이 아닌 결과물과 기술 방향 중심 |
| **Generator** | React + Vite + FastAPI + SQLite/PostgreSQL로 구현 | 스프린트 단위 작업, 자체 평가 후 QA에 전달 |
| **Evaluator** | Playwright MCP로 실제 사용자처럼 테스트 | 각 스프린트 전 "스프린트 계약서"로 완료 기준 협상 |

> 에이전트 간 통신은 **파일 기반** — 명세 충실성 유지 + 구현 자유도 확보

### Solo Run vs Full Harness 비교

| 항목 | Solo Run | Full Harness |
|------|----------|-------------|
| **소요 시간** | 20분 | 6시간 |
| **비용** | $9 | $200 |
| **결과** | 낭비된 레이아웃 공간, 경직된 워크플로우, 깨진 핵심 게임 메카닉 | 16개 기능, 10 스프린트, 스프라이트 애니메이션, 사운드, AI 생성, 공유 기능 |
| **품질** | UI 가이드 없음, 문제 표시 없음 | 일관된 시각 아이덴티티, 사용 가능한 에디터, 작동하는 게임 |

### QA 프로세스의 도전

- 초기: 합법적 이슈 식별하나 **합리화**하여 수용
- 표면적 테스트만 수행, 엣지 케이스 미탐색
- 반복 조정 후에도 잔존: 미세한 레이아웃 이슈, 비직관적 인터랙션, 중첩 기능의 미발견 버그

## 04. 하네스 진화

### 핵심 원칙

> "하네스의 모든 컴포넌트는 모델이 혼자 할 수 없다는 가정을 인코딩한다. 그 가정은 스트레스 테스트할 가치가 있다."

### Opus 4.6으로 단순화

| 이전 (Opus 4.5) | 이후 (Opus 4.6) |
|-----------------|-----------------|
| 스프린트 분해 필수 | 스프린트 없이 일관된 작업 가능 |
| Evaluator 필수 | 단순 태스크에서는 오버헤드 |
| 복잡한 스캐폴딩 | 불필요한 구성요소 제거 |

**판단 기준**: Evaluator의 필요성은 태스크 복잡도 대비 모델 능력의 함수

### DAW (Digital Audio Workstation) 사례

**프롬프트**: "브라우저에서 Web Audio API를 활용한 완전한 기능의 DAW를 만들어라"

| 단계 | 소요 시간 | 비용 |
|------|----------|------|
| Planner | 4.7분 | $0.46 |
| Build 1 | 2시간 7분 | $71.08 |
| QA 1 | 8.8분 | $3.24 |
| Build 2 | 1시간 2분 | $36.89 |
| QA 2 | 6.8분 | $3.09 |
| Build 3 | 10.9분 | $5.88 |
| QA 3 | 9.6분 | $4.06 |
| **합계** | **~4시간** | **$124.70** |

- Builder가 **2시간 넘게** 스프린트 분해 없이 일관성 유지
- QA가 핵심 갭 식별: 클립 드래그 불가, 악기 UI 패널 미구현, 이펙트 시각화 미흡
- 최종 결과: 프롬프팅만으로 간단한 곡 제작 가능

## 05. 핵심 교훈

### 3가지 핵심 레슨

1. **실험 우선**: 실제 문제에 대상 모델로 실험하고, 트레이스 기반으로 성능 튜닝
2. **분해와 전문화**: 복잡한 태스크를 분해하고 특화된 에이전트를 각 측면에 적용
3. **새 모델 출시 시 재검토**: 비핵심 컴포넌트를 제거하고, 더 큰 능력을 가능케 하는 조각을 추가

> "모델이 발전해도 흥미로운 하네스 조합의 공간은 축소되지 않는다. 대신 이동한다. AI 엔지니어의 흥미로운 작업은 다음 새로운 조합을 계속 찾는 것이다."
