# Bug Hunter — 근본 원인 분석 + 병렬 수정 + E2E 검증 통합 스킬

버그를 발견했을 때, 표면적 증상만 고치면 같은 문제가 반복된다. 이 스킬은 **근본 원인(Root Cause)**까지 파고들어 확실히 해결하고, Codex in Chrome으로 실제 사용자 시점에서 검증한 뒤, 기획서와 리포트를 남기는 전체 프로세스를 안내한다.

## 전체 흐름

```
Phase 1: 버그 수집 & 분류
  ↓
Phase 2: 근본 원인 분석 (병렬 에이전트)
  ↓
Phase 3: 수정 구현
  ↓
Phase 4: Codex in Chrome E2E 검증 + GIF/스크린샷
  ↓
Phase 5: 기획서 & 완료 리포트 생성
```

---

## Phase 1: 버그 수집 & 분류

### 1-1. 버그 목록 작성
버그 정보를 다음 형식으로 정리한다:

```markdown
| # | 버그 설명 | 증상 | 심각도 | 영향 범위 |
|---|----------|------|--------|----------|
| B-01 | ... | ... | HIGH/MEDIUM/LOW | ... |
```

### 1-2. 심각도 기준

| 심각도 | 기준 | 예시 |
|--------|------|------|
| **CRITICAL** | 핵심 기능 완전 차단, 데이터 손실 위험 | 참가 신청 불가, DB 무결성 파괴 |
| **HIGH** | 주요 기능 동작 불가 | API 500 에러, 기능 사용 불가 |
| **MEDIUM** | 기능은 동작하나 UX 저하 또는 데이터 부정확 | 타입 불일치, UI 깜빡임 |
| **LOW** | 콘솔 노이즈, 개발 환경 전용 | 로그 스팸, HMR 경고 |

### 1-3. 우선순위 결정
심각도 × 수정 난이도 × 사용자 영향도를 종합하여 처리 순서를 정한다. CRITICAL/HIGH를 먼저 처리하되, "수정 난이도 하"인 것은 즉시 처리하여 빠른 성과를 낸다.

## Phase 2: 근본 원인 분석

이 단계가 가장 중요하다. 증상 뒤에 숨은 진짜 원인을 찾아야 재발을 막을 수 있다.

### 2-1. 5 Whys 분석법

각 버그에 대해 최소 3단계 "왜?"를 반복한다:

```
증상: API가 500 에러를 반환한다
→ Why 1: 동적 임포트가 런타임에서 실패한다
→ Why 2: 경로 별칭(@shared/)이 서버 번들에서 해석되지 않는다
→ Why 3: tsconfig paths가 런타임 해석기에 전달되지 않는다
→ 근본 원인: 동적 임포트 대신 정적 임포트를 사용해야 한다
```

### 2-2. 병렬 에이전트 팀 구성

독립적인 버그들은 병렬 에이전트로 동시에 분석한다. 단일 메시지에서 여러 Agent 호출을 보내면 된다.

**에이전트 역할 분배 패턴:**

```
에이전트 1 (서버 분석): 서버 코드, API, DB 관련 버그
에이전트 2 (클라이언트 분석): React, UI, 상태관리 관련 버그
에이전트 3 (통합 분석): 서버-클라이언트 연동, 인증, WebSocket 관련 버그
```

**에이전트 프롬프트 템플릿:**

```
버그 [B-XX] 근본 원인을 분석해줘.

증상: [증상 설명]
관련 파일: [파일 경로들]

분석 순서:
1. 관련 파일을 읽고 코드 흐름 추적
2. 에러 발생 조건 파악 (어떤 입력/상태에서 발생하는지)
3. 5 Whys로 근본 원인 도출
4. 수정 방안 제시 (최소 변경 원칙)

결과를 다음 형식으로 보고해줘:
- 근본 원인: [한 문장]
- 영향 받는 코드: [파일:라인]
- 수정 방안: [구체적 코드 변경 내용]
- 부수 효과: [다른 기능에 미치는 영향]
```

### 2-3. 근본 원인 문서화

분석 결과를 버그별로 정리한다:

```markdown
### B-XX: [버그 제목]

**근본 원인**: [PostgreSQL NULL 비교 특성으로 인한 유니크 제약 우회]
**코드 위치**: `파일경로:라인번호`
**발생 조건**: [어떤 상황에서 트리거되는지]
**수정 방안**: [구체적 변경 내용]
**연쇄 영향**: [이 수정이 다른 기능에 미치는 영향]
```

## Phase 3: 수정 구현

### 3-1. 수정 원칙
- **최소 변경**: 버그 수정에 필요한 최소한의 코드만 변경한다
- **논리적 단위 커밋**: 버그별 또는 관련 버그 묶음별로 커밋한다
- **기존 패턴 준수**: 프로젝트의 코딩 스타일과 패턴을 따른다

### 3-2. 병렬 수정 (독립적 버그인 경우)
서로 다른 파일을 건드리는 독립 버그들은 병렬 에이전트로 동시에 수정한다.

**주의**: 같은 파일을 수정하는 버그들은 반드시 순차적으로 처리한다.

### 3-3. 수정 후 단위 테스트
- `npx vitest run` 또는 관련 테스트 파일만 실행하여 기존 테스트 통과 확인
- `npx tsc --noEmit`으로 타입 체크

## Phase 4: Codex in Chrome E2E 검증 + GIF/스크린샷

실제 사용자 시점에서 버그가 해결되었는지 확인한다. **Codex in Chrome MCP** 도구를 사용한다.

### 4-1. SSO 인증 세션 확인

E2E 검증 전에 인증 세션이 필요한 경우:

1. **현재 탭 확인**: `mcp__claude-in-chrome__tabs_context_mcp`로 열린 탭 확인
2. **로그인 상태 확인**: 대상 페이지로 이동하여 로그인 여부 확인
3. **로그인 필요 시**: 사용자에게 안내
   ```
   SSO 로그인 세션이 필요합니다.
   Chrome에서 https://daker.ai (또는 http://localhost:5000)에 접속하여
   DACON SSO 로그인을 완료해주세요.
   로그인 후 다시 요청해주시면 E2E 검증을 진행하겠습니다.
   ```
4. **로그인 상태 확인**: 페이지 이동 후 `mcp__claude-in-chrome__get_page_text`로 인증 상태 확인

### 4-2. Codex in Chrome MCP로 검증

다음 도구들을 사용하여 실제 브라우저에서 검증한다:

| 단계 | MCP 도구 | 용도 |
|------|----------|------|
| 1 | `mcp__claude-in-chrome__tabs_context_mcp` | 현재 탭 상태 확인 |
| 2 | `mcp__claude-in-chrome__navigate` | 버그 재현 페이지로 이동 |
| 3 | `mcp__claude-in-chrome__read_page` | 현재 페이지 DOM/상태 확인 |
| 4 | `mcp__claude-in-chrome__computer` | 클릭, 입력 등 사용자 동작 시뮬레이션 |
| 5 | `mcp__claude-in-chrome__form_input` | 폼 필드 입력 |
| 6 | `mcp__claude-in-chrome__read_console_messages` | 콘솔 에러 확인 |
| 7 | `mcp__claude-in-chrome__read_network_requests` | API 응답 상태 확인 |
| 8 | `mcp__claude-in-chrome__javascript_tool` | JS 실행으로 상태 확인 |

### 4-3. GIF 녹화 (검증 과정 캡처)

멀티 스텝 검증 시나리오에서는 **GIF 녹화**를 사용하여 전체 과정을 기록한다:

```
GIF 녹화 절차:
1. mcp__claude-in-chrome__gif_creator 호출하여 녹화 시작
2. 각 동작 전후로 추가 프레임 캡처 (부드러운 재생을 위해)
3. 의미 있는 파일명으로 저장:
   - docs/plans/screenshots/YYYYMMDD-bug-BXX-verification.gif
   - docs/plans/screenshots/YYYYMMDD-bug-BXX-before-after.gif
```

**GIF 녹화가 적합한 경우:**
- 여러 단계를 거쳐 재현되는 버그
- 수정 전/후 동작 비교
- UI 인터랙션이 포함된 검증 시나리오

### 4-4. 스크린샷 저장

정적 캡처는 `docs/plans/screenshots/` 디렉토리에 저장한다:

```
docs/plans/screenshots/
├── YYYYMMDD-bug-BXX-before.png    (수정 전)
├── YYYYMMDD-bug-BXX-after.png     (수정 후)
├── YYYYMMDD-bug-BXX-verified.png  (검증 완료)
└── YYYYMMDD-bug-BXX-verification.gif (GIF 녹화)
```

### 4-5. 검증 결과 기록

```markdown
| # | 시나리오 | 결과 | 캡처 | 비고 |
|---|---------|------|------|------|
| T-01 | [검증 시나리오] | ✅ PASS / ❌ FAIL | [파일명] | [추가 메모] |
```

## Phase 5: 기획서 & 완료 리포트 생성

### 5-1. 버그 조사 기획서

**파일 경로**: `docs/plans/YYYY-MM-DD-{주제}_버그조사_기획.md`

`references/bug-plan-template.md`를 참고하여 작성한다.

### 5-2. 수정 완료 리포트

**파일 경로**: `docs/plans/YYYY-MM-DD-{주제}_버그수정_완료리포트.md`

`references/bug-report-template.md`를 참고하여 작성한다.

### 5-3. bug-list.md 업데이트

프로젝트에 `docs/bug-list.md`가 있으면 처리한 버그의 상태를 업데이트한다:

```markdown
| # | 버그 | 심각도 | 상태 | 수정일 | 리포트 |
|---|------|--------|------|--------|--------|
| B-XX | [설명] | HIGH | ✅ 수정됨 | YYYY-MM-DD | [링크] |
```

파일이 없으면 새로 생성하지 않는다 (필요 시 사용자에게 생성 여부 확인).

## 병렬 실행 전략

### 어떤 작업을 병렬로 할 수 있는가

| 병렬 가능 | 순차 필수 |
|-----------|----------|
| 독립 버그의 근본 원인 분석 | 같은 파일 수정하는 버그들 |
| 서버/클라이언트 분리된 수정 | 테스트 → 수정 → 재테스트 |
| 여러 페이지의 E2E 검증 | DB 스키마 변경 → API 수정 |
| 기획서/리포트 동시 작성 | 인증 세션 확보 → E2E 검증 |

### 에이전트 팀 구성 예시

```
[메인 에이전트] → 전체 조율, 기획서/리포트 작성

[에이전트 A - 서버] → B-01, B-03 근본 원인 분석 + 수정
[에이전트 B - 클라이언트] → B-02, B-04 근본 원인 분석 + 수정
[에이전트 C - E2E 검증] → Codex in Chrome으로 수정 결과 확인 + GIF/스크린샷
```

## 핵심 원칙

1. **증상이 아닌 원인을 고쳐라**: "500 에러가 나서 try-catch를 추가했다"는 수정이 아니다. 왜 500이 나는지 찾아서 고쳐야 한다.
2. **재현 → 분석 → 수정 → 검증 순서**: 버그를 재현할 수 없으면 수정도 검증도 불가능하다.
3. **연쇄 영향을 항상 확인하라**: 한 곳을 고치면 다른 곳이 깨질 수 있다. 수정 후 전체 테스트를 반드시 돌려라.
4. **GIF/스크린샷은 증거다**: "수정했다"는 말보다 수정 전/후 캡처가 훨씬 설득력 있다.
