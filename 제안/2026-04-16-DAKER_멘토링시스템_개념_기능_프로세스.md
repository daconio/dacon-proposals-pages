# DAKER 멘토링 시스템 — 개념·기능·프로세스

> **작성일**: 2026-04-16
> **대상**: 제안서용 기능 소개 자료
> **범위**: AI 오피스아워 멘토링 + 휴먼 멘토 1:1 예약/협상 시스템

---

## 1. 개념 (Overview)

DAKER 멘토링 시스템은 해커톤·경진대회 환경에서 **참가팀의 사고(Thinking)와 실행(Execution)을 함께 끌어올리는 이중 멘토링 구조**다.

| 구분 | AI 멘토링 (Office Hours) | 휴먼 멘토 예약 (Mentor Booking) |
|------|--------------------------|--------------------------------|
| 목적 | 24/7 Forcing Question으로 팀의 사고 압박 | 전문가와의 1:1 집중 미팅 운영 |
| 주체 | LLM 기반 AI 멘토 | 해커톤 배정 휴먼 멘토 |
| 트리거 | 참가자가 언제든 채팅 진입 | 팀장 예약/제안 → 멘토 확정 |
| 컨텍스트 | 팀 스킬·가용시간·프로필 + 해커톤 목표·평가기준 스냅샷 | 해커톤 기간·쿼터·모드·장소 설정 |
| 산출물 | 체크포인트 기반 대화 로그 + 리포트 | 미팅 후 구조화 리포트 (요약/피드백/액션아이템) |

**핵심 가치**
1. **맥락 주입(Context-Aware)**: 멘토(AI·휴먼)가 팀·대회의 실제 정보를 항상 인용
2. **강한 질문(Forcing Question)**: VC 오피스아워 스타일 6대 질문으로 팀의 가설을 실전 검증
3. **자율 협상(Hybrid Booking)**: 멘토 오픈 슬롯 즉시 예약 + 슬롯 밖 시간 제안의 2-way 협상
4. **책임성(Accountability)**: 쿼터·노쇼·cutoff 정책으로 운영 리스크 관리
5. **리포트 자동화**: 모든 세션이 구조화된 리포트로 적재 → 운영팀 사후분석 자산
6. **캘린더 기반 상호 스케줄링**: 멘토와 팀장이 **동일한 주간 캘린더 뷰**를 공유. 멘토는 드래그로 가용시간을 공개하고, 팀장은 같은 뷰에서 즉시 예약 또는 슬롯 밖 시간 제안. 왕복 연락 없이 플랫폼 안에서 협상 완결.

## 2. AI 멘토링 (Office Hours 모드)

### 2.1 개념
VC 오피스아워 방식(gstack 스타일)으로 재설계한 AI 대화. 단순 Q&A 챗봇이 아니라, 팀의 속성과 해커톤의 평가기준을 **컨텍스트 스냅샷**으로 먼저 파악한 뒤, 6가지 Forcing Question을 단계에 맞게 제시한다.

### 2.2 6대 Forcing Question

| # | 영문 | 한국어 번안 | 의도 |
|---|------|-------------|------|
| Q1 | Demand Reality | 수요의 실재 | 누가 실제로 원하는가 |
| Q2 | Status Quo | 현재 대안·비용 | 지금은 어떻게 해결 중인가 |
| Q3 | Desperate Specificity | 구체적 타겟 | 누가 "절박하게" 필요한가 |
| Q4 | Narrowest Wedge | **최소 검증 단위** | 가장 좁게 증명할 범위는 |
| Q5 | Observation & Surprise | 관찰과 의외성 | 관찰로 얻은 비상식 발견은 |
| Q6 | Future-Fit | 3년 적합성 | 3년 뒤에도 유효한가 |

### 2.3 스테이지 라우팅
해커톤 체크포인트별로 허용 질문이 달라진다.

| 체크포인트 | 단계 | 허용 Q |
|-----------|------|--------|
| CP1 | 탐색 | Q1 → Q2 → Q3 |
| CP2 | 기획 | Q2 → Q4 |
| CP3 | 개발 | Q4 → Q5 |
| CP4 | 발표 | Q5 → Q6 |
| 자유 채팅 | 컨텍스트 추정 | 선택 |

### 2.4 톤 (Tone) 선택
운영자가 해커톤 성격에 맞게 3가지 톤 중 선택.

| tone | 특성 | 사용 예 |
|------|------|---------|
| `yc-hard` (기본) | 직설, Red flag 즉시 지적, calibrated acknowledgment 짧게 | 대학·성인 해커톤 |
| `yc-soft` | Forcing question 유지, 칭찬 비중↑, 경어 강화 | 기업 교육형 |
| `coach` | 푸시백 최소, 선택지 제시 | 청소년·입문자 |

### 2.5 컨텍스트 스냅샷 (세션 생성 시 DB 저장)
- **해커톤**: title, objectives, evaluationCriteria, 기간
- **팀**: 이름, 솔로 여부, 팀원별 {displayName, skills, bio, portfolio, weeklyHours}, 총 가용시간
- **이전 리포트**: 체크포인트별 직전 세션 요약
- **signalCount**: 팀 속성 시그널이 2 미만이면 "Builder Mode"로 폴백(일반 조언)

### 2.6 대화 품질 규칙
- 첫 응답에 **팀원 닉네임 ≥1 + 스킬 ≥1 + 평가기준 ≥1** 반드시 인용
- 1 응답 = 1 주질문(질문부호 ≤ 2)
- 유저가 "just do it" 류 거부 신호 → Escape Hatch(남은 2개만 묻고 요약)
- 금칙어 0회: delve, crucial, robust, comprehensive, pivotal, landscape, furthermore 등

## 3. 휴먼 멘토 예약 (Mentor Booking)

### 3.1 개념
AI 멘토링이 커버하지 못하는 **도메인 전문가 1:1 미팅**을 운영 가능한 형태로 시스템화. 관리자·멘토·팀장 3자 역할이 명확히 분리되며, 온/오프라인 혼합 운영을 지원한다.

### 3.2 참여자 역할

| 역할 | 권한 |
|------|------|
| **관리자** | 해커톤별 예약 정책 설정(기간, 한도, 모드, 기본 장소, cutoff) |
| **멘토** | 슬롯 대량 오픈, 공개범위 지정, 제안 승인/거절, 리포트 작성 |
| **팀장** | 슬롯 즉시 예약, 슬롯 밖 시간 제안, 취소, 리포트 열람 |
| **팀원** | 팀 예약 열람(CTA 비활성) |

### 3.3 관리자 설정 항목
- `enabled` — 멘토 예약 기능 on/off
- `windowStart / windowEnd` — 예약 가능 기간 (KST)
- `maxDurationPerMeetingMin` — 1회 최대 미팅 시간
- `maxTotalPerTeamMin` — 팀당 누적 한도
- `allowedModes` — `online`, `offline`, 또는 둘 다
- `defaultOfflineLocation` — 해커톤 기본 오프라인 장소
- `cancelCutoffMin` — 취소 가능 마감 (기본 60분 전)

### 3.4 멘토 슬롯 모델
- `mode`: online | offline
- `visibility`: **assigned**(배정 팀만 예약 가능) | **public**(전체 공개)
- `status`: open → booked → (canceled / completed)
- `source`: mentor(직접 오픈) | proposal(팀 제안 승인으로 생성)

**대량 오픈 UX**: 주간 그리드(7일 × 30분 셀)에서 드래그로 N개 슬롯 프리뷰 → 일괄 생성, 개별 편집은 리스트 뷰.

### 3.5 하이브리드 협상 방식

| 경로 | 흐름 |
|------|------|
| **즉시 예약** | 팀장이 오픈 슬롯 클릭 → 즉시 확정 (슬롯당 1팀) |
| **시간 제안** | 팀장이 슬롯 밖 시간 제안 → 멘토 승인 시 슬롯+예약 원자적 생성 |

슬롯 즉시 확정은 빠른 운영을, 제안 승인은 유연한 일정 조율을 동시에 제공한다.

### 3.6 캘린더 기반 상호 스케줄링 (핵심 UX)

멘토와 팀장이 **동일한 주간 캘린더 뷰**를 공유하며 양방향으로 일정을 조율한다. 별도 메일/메신저 교환 없이 플랫폼 안에서 협상이 완결된다.

#### 공통 컴포넌트 — `WeeklySlotGrid`
- **구조**: 7일 × 30분 셀 그리드 (기본 표시 시간 10:00–22:00, 설정 가능)
- **모드 분기**:
  - `mode='edit'` (멘토): 드래그로 다중 셀 선택 → 프리뷰 분할 → 일괄 슬롯 생성
  - `mode='view'` (팀장): 오픈 슬롯 클릭으로 예약, 빈 셀 클릭으로 시간 제안 트리거
- **드래그 로직**: 같은 날 범위만 허용, 기존 슬롯 있는 셀에서는 드래그 시작 차단, 연속 블록을 `defaultDurationMin`(30/60/90분) 단위로 자동 분할
- **키보드 접근성**: 방향키 포커스(`data-cell-id` 기반), WCAG AA, 한국어 aria-label

#### 멘토 측 (Producer) — 캘린더에 가용시간 입력
1. 멘토 페이지 → "슬롯 관리" 탭
2. 주간 그리드에서 드래그 → N개 슬롯 프리뷰
3. 모드(online/offline)·공개범위(`assigned`/`public`) 지정
4. `POST /api/mentor/hackathons/:id/slots:bulk` 일괄 생성
5. 리스트 뷰(`SlotList`)에서 개별 편집·삭제 (예약된 슬롯은 장소·링크만 수정)

#### 팀장 측 (Consumer) — 같은 캘린더에서 예약/제안
1. "멘토 찾기" 탭 → 멘토 카드 선택 → 상세 페이지 진입
2. `WeeklySlotGrid view 모드`에서 멘토의 오픈 슬롯을 색상 구분으로 즉시 파악
   - 본인 팀 예약 슬롯은 `currentRaidId` 일치로 구분 표시
   - 타 팀 예약은 **busy 마스킹**(`MaskedBooking`)만 보여 프라이버시 보호
3. 경로 A — **오픈 슬롯 클릭** → `SlotDetailPopover`에서 모드/장소/링크 확인 → 즉시 예약
4. 경로 B — **빈 셀 클릭** → `ProposalDialog` 열림
   - datetime-local 입력을 브라우저 로컬 TZ → ISO 변환
   - 모드 선택 + 메시지 첨부
   - `POST /api/team/:raidId/proposals` 제안 제출

#### 멘토의 제안 응답 (비동기)
- "제안함" 탭에 pending 목록 노출
- 승인 시 트랜잭션 안에서 `mentor_slots`(source='proposal') + `mentor_bookings` 원자적 생성
- 거절 시 응답 메시지 첨부, 팀에게 알림 발송

#### 캘린더 UI 상태 가시화
- **BookingCard 채움단계 배지**: confirmed=solid / pending=tint / canceled=muted / no_show=outline
- **QuotaBadge**: 잔여 시간 ≤30분이면 경고 톤 자동 전환
- **동시성 충돌**: 슬롯 경쟁 실패 시 409 → 토스트 + 캘린더 자동 새로고침
- **cutoff 시각화**: 취소 cutoff가 지난 예약은 "노쇼 위험" 시각 단서

#### 상호 스케줄링의 설계 의미
- 멘토와 팀이 **하나의 주간 뷰를 공통 언어로 공유** → "언제 가능하세요?" 왕복 제거
- 슬롯(즉시확정) + 제안(협상)이 같은 캘린더 위에서 동작 → UX 분리 부담 없음
- 타 팀 예약을 busy로만 노출 → 경쟁 압력 유도하면서 프라이버시 유지
- 드래그 기반 대량 생성 + 리스트 개별 편집 → 멘토의 운영 피로 최소화

### 3.6 쿼터 & 노쇼 정책
- `used = Σ duration WHERE status IN ('confirmed','completed','no_show')`
- `remaining = maxTotalPerTeamMin - used`
- `canceled`는 회복, **`no_show`는 미회복** (어뷰즈 방지)
- 취소 cutoff 이후 취소 = `no_show_flag=true`
- UI에서 잔여 쿼터 배지(`QuotaBadge`) 상시 노출

### 3.7 알림 5종 (in-app + 이메일)

캘린더 협상이 비동기로 진행되므로, 상태 전이마다 자동 알림이 캘린더 밖에서도 흐름을 이어준다.

- `mentor_booking.created` — 예약 생성
- `mentor_booking.canceled` — 예약 취소
- `mentor_booking.reminder_30m` — 미팅 30분 전 자동 리마인더
- `mentor_proposal.submitted` — 제안 제출(멘토에게)
- `mentor_proposal.responded` — 제안 응답(팀에게)

### 3.9 미팅 후 리포트
멘토가 미팅 종료 후 작성:
- **summary** — 논의 요약
- **feedback** — 팀에 대한 피드백
- **actionItems** — 다음 액션 아이템

기존 `mentoring_sessions` / `mentoring_reports` 테이블을 `kind='human'`, `reportType='human_meeting'`으로 재활용 → AI·휴먼 리포트 단일 뷰.

## 4. 프로세스 (End-to-End Flow)

### 4.1 AI 멘토링 세션 흐름

```
참가팀 채팅 진입
  ↓
서버: 팀·해커톤 컨텍스트 스냅샷 생성 → mentoring_sessions.contextSnapshot 저장
  ↓
첫 응답: 팀원 닉네임 + 스킬 + 평가기준 인용 + 체크포인트별 Q 제시
  ↓
유저 응답 → Q 라우터 (stage + 답변 품질 판단)
  ↓
다음 Q 이동 / 리프레임 요청 / Escape Hatch
  ↓
세션 종료 시 구조화 리포트 자동 생성
```

### 4.2 휴먼 멘토 예약 전체 흐름

```
[1] 관리자: 해커톤 페이지 → 멘토 예약 설정 (기간·한도·모드·cutoff)
       ↓
[2] 멘토: 주간 그리드 드래그 → 슬롯 N개 bulk 생성 (visibility 선택)
       ↓
[3] 팀장: 멘토 상세 페이지에서
   (a) 오픈 슬롯 클릭 → 즉시 확정
   (b) 슬롯 밖 시간 제안 → 멘토 승인 대기
       ↓
[4] 알림 발송 (in-app + 이메일)
       ↓
[5] 미팅 30분 전 자동 리마인더
       ↓
[6] 미팅 진행 (온라인 링크 또는 오프라인 장소)
       ↓
[7] 멘토: 리포트 작성 (요약/피드백/액션아이템)
       ↓
[8] 팀장: 리포트 열람 → 다음 체크포인트 AI 멘토링에 반영
```

### 4.3 즉시 예약 트랜잭션 (동시성 안전)
1. `SELECT ... FOR UPDATE on mentor_slots`
2. 검증 순서: status='open' → config enabled/window → visibility 규칙 → 팀 쿼터 → 팀장 권한
3. `mentor_bookings` insert → `mentor_slots.status='booked'`
4. 알림 + 이메일 트랜잭션 내 삽입
5. 실패 시 409 `SLOT_ALREADY_BOOKED` + 자동 refetch

### 4.4 취소 & 노쇼 판정
```
nowKST = 현재 시각
cutoffTime = slot.start_at - cancelCutoffMin
if nowKST < cutoffTime:
  booking.status='canceled', no_show_flag=false   # 쿼터 회복
else:
  booking.status='canceled', no_show_flag=true    # 쿼터 미회복
mentor_slots.status='canceled'                    # 재오픈 안 함
```

### 4.5 AI × 휴먼 연결 고리
- 팀장이 휴먼 멘토 리포트를 읽은 뒤 AI 멘토링 진입 시, AI는 해당 리포트의 액션 아이템을 다음 Forcing Question의 맥락으로 활용
- 반대로 AI 멘토링에서 도출한 "최소 검증 단위(Q4)"를 휴먼 멘토와의 미팅 주제로 제안 가능
- **단일 리포트 스트림**으로 팀 성장 궤적 추적

## 6. 설계 원칙

### 6.1 컨텍스트 우선
AI든 휴먼이든 "팀이 무엇을 하고 있고, 평가기준이 무엇인지" 먼저 본다. 일반론 금지.

### 6.2 No Overengineering
- 그룹 멘토링(슬롯당 다팀) 미지원
- 외부 캘린더 연동(.ics, Google Calendar) 1차 제외
- 멀티 타임존 미지원 (KST 고정)
- 실시간 화상 내장 없음(외부 링크만)

### 6.3 어뷰즈 방지
- 노쇼 쿼터 미회복
- 슬롯 취소 시 자동 재오픈 금지 (rush 방지, 멘토가 재생성)

### 6.4 운영 가시성
- 쿼터 잔여 배지 상시 노출
- 모든 상태 전이는 알림 발송
- 리포트 미작성 미팅은 운영 대시보드에서 추적

### 6.5 디자인 시스템 준수
- 3-Color 원칙(Primary #0053db / Foreground #2a3439 / Card #ffffff)
- No-Line (1px 보더 금지, 배경 톤 전환으로 분리)
- 상태 배지 채움단계: confirmed=solid / pending=tint / canceled=muted / no_show=outline
- 카드 그리드 통일: `grid-cols-1 min-[600px]:grid-cols-2 xl:grid-cols-3`
- WCAG AA, Plus Jakarta Sans, 한국어 UI

## 7. 기대 효과

| 관점 | 효과 |
|------|------|
| **참가팀** | AI가 24/7 강한 질문으로 사고를 압박, 휴먼 멘토는 도메인 검증에 집중. 체크포인트별 누적 리포트로 팀 성장 궤적 시각화 |
| **멘토** | 슬롯 대량 오픈·공개범위 태깅으로 시간 운영 효율화. 제안/승인 비동기 협상으로 일정 부담 완화 |
| **운영사(DAKER)** | 쿼터·노쇼·cutoff 정책으로 운영 리스크 최소화. 모든 대화/미팅이 구조화된 리포트로 자산화되어 사후 분석 가능 |
| **주관기관** | AI 멘토링으로 1:1 멘토링 병목 해소, 휴먼 멘토는 핵심 심사/검증에만 집중. 교육 효과 + 운영 비용 최적화 동시 달성 |
