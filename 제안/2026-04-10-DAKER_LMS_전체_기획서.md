## 1. 개요

### 1.1 비전

DAKER LMS는 **AI 기반 맞춤형 학습 관리 시스템**으로, 기업·대학·공공기관의 AI/데이터 교육을 위한 올인원 플랫폼이다. 단순 콘텐츠 전달을 넘어 **AI 교재 자동 생성**, **실시간 학습 분석**, **게이미피케이션**, **팀 학습 연계**를 통합하여 학습 효과를 극대화한다.

2026년 4월, **Hybrid Plugin LMS 4-Layer 아키텍처**를 도입하여 스테이지 타입 확장성, 이벤트 기반 분석, 통합 성적부 등 차세대 LMS 기능을 구현하고 있으며, **학습 그룹 관리**, **일괄 등록 시스템**, **경량 출결 시스템**을 완성하여 대규모 조직 교육 운영 인프라를 갖추었다.

### 1.2 핵심 가치

| 가치 | 설명 |
|------|------|
| **AI-Native 교재 생성** | Gemini + GPT 상호 검수 파이프라인으로 교재를 자동 생성. 관리자의 교재 제작 부담을 90% 이상 절감 |
| **플러그인 기반 확장** | 새로운 스테이지 유형을 **1파일 + 레지스트리 등록 2줄**로 추가. 기존 10개 파일 수정 → 사실상 1시간 내 확장 |
| **xAPI 이벤트 스트림** | 12종 Verb × 5종 Object Type의 학습 이벤트를 실시간 수집. 이탈 분석, 완료 퍼널 등 7대 분석 지표 제공 |
| **통합 성적부 (Gradebook)** | 퀴즈·프롬프트·플래시카드의 시도 데이터를 교재/트랙 단위로 통합 집계. 관리자·학습자 모두 성적 확인 가능 |
| **게이미피케이션** | Apple Watch 스타일 활동 링, 스트릭, 배지, 리더보드로 학습 동기 부여 |
| **학습 그룹 관리** | 그룹 단위 수강 등록, 기간별 권한 관리, 일괄 등록(최대 1,000명) 4단계 마법사로 대규모 교육 운영 |
| **경량 출결 시스템** | 세션 로그 기반 자동 감지 + 관리자 수동 편집. 그리드 UI로 출결 현황 한눈에 관리, 학습자 셀프 대시보드 제공 |
| **해커톤/Raid 연계** | DAKER의 해커톤·Raid 플랫폼과 유기적으로 연결된 학습 트랙 시스템 |

| **관리자 (Admin)** | 교재 생성/관리, 학습 분석, 수강생 관리, 트랙 구성, 목표 설정, Gradebook 성적 관리 |

## 2. Hybrid Plugin LMS 4-Layer 아키텍처

### 2.1 왜 "Hybrid Plugin"인가

해외 LMS 플랫폼(Moodle, Canvas, Coursera, Khan Academy 등)과 국내 플랫폼을 리서치한 결과, 두 가지 접근법이 존재했다:

| 접근법 | 장점 | 단점 | 대표 |
|--------|------|------|------|
| **모놀리식** | 안정적, 일관된 UX | 타입 추가 시 대규모 수정 필요 | Coursera, Khan Academy |
| **플러그인 기반** | 높은 확장성, 외부 개발자 참여 | 복잡한 인터페이스, 버전 관리 | Moodle, Canvas |

DAKER는 단일팀 단일 코드베이스이므로 외부 플러그인 공개 필요 없이, **내부 개발 속도 향상**을 위한 Hybrid 전략을 채택했다:
- 플러그인 인터페이스는 **내부용** (StagePlugin 계약)
- 배포는 **단일 번들** (기존 빌드/배포 파이프라인 유지)
- 기존 동작을 **한 줄도 깨지 않으면서** 플러그인 구조를 덧씌움

### 2.2 4-Layer 구조

```
┌─────────────────────────────────────────────────────────────┐
│ Layer 4 — 표준 호환 (SCORM/xAPI Export)                      │ ⏭️ 생략
├─────────────────────────────────────────────────────────────┤
│ Layer 3 — Assessment (Gradebook + 문제은행 + 과제)           │ 🔄 Phase 3-A 진행중
│   └─ Gradebook 서비스: plugin.score() → 통합 성적 집계       │
├─────────────────────────────────────────────────────────────┤
│ Layer 2 — 학습 이벤트 스트림 (xAPI 경량)                      │ ✅ Phase 1 완료
│   └─ learning_events 테이블 + 12 verbs + 7대 분석 지표       │
├─────────────────────────────────────────────────────────────┤
│ Layer 1 — Stage Plugin Core (스테이지 타입 확장 인프라)       │ ✅ Phase 2 완료
│   └─ 6종 플러그인 + 서버-클라이언트 대칭 레지스트리          │
└─────────────────────────────────────────────────────────────┘
                          ↑
                 기존 DAKER Learning Domain
              (교재/트랙/세션/스트릭/배지/AI생성)
```

### 2.3 Layer 간 의존성 및 데이터 흐름

```
사용자가 퀴즈 제출
  → LearningQuizViewer (Layer 1 quiz 플러그인)
  → plugin.toEvents() → eventStream.recordBatch() → learning_events (Layer 2)
  → plugin.score() → gradebookService 집계 (Layer 3)
  → 학습자 성적 뷰 / 관리자 Gradebook 대시보드
```

| Layer | 역할 | 산출물 | 다른 Layer와의 관계 |
|-------|------|--------|--------------------|
| **Layer 1** | 스테이지 타입 플러그인 | `StagePlugin` 인터페이스, 레지스트리, 6종 플러그인 | Layer 2에 `toEvents()`로 이벤트 주입, Layer 3에 `score()` 제공 |
| **Layer 2** | 학습 이벤트 단일 소스 | `learning_events` 테이블, 분석 대시보드 | Layer 1의 `toEvents()` 결과를 수집, Layer 3의 원시 데이터 |
| **Layer 3** | 평가·성적 통합 | Gradebook 서비스, 문제 은행, 과제 | Layer 2 이벤트 집계 + Layer 1의 `score()` 호출 |
| **Layer 4** | 외부 표준 호환 | SCORM/xAPI export | 생략 (필요 시 Layer 2 변환으로 저비용 도입) |

### 2.4 Phase 순서 및 이유

```
Phase 1 (Layer 2) ✅ 완료
         ↓
Phase 2 (Layer 1) ✅ 완료 ──→ toEvents() 계약으로 Phase 1 수집 훅 자동화
         ↓
Phase 3 (Layer 3) 🔄 진행중 ──→ 이벤트 + score() 조합으로 Gradebook 자연 구성
```

**Phase 1을 먼저 한 이유**: Layer 2의 이벤트 스트림이 없으면 Layer 1의 `toEvents()`는 저장할 곳이 없고, Layer 3의 Gradebook은 집계할 단일 소스가 없다.

## 3. 콘텐츠 계층 구조

DAKER LMS는 **3-Tier 계층 구조**로 학습 콘텐츠를 구성한다.

### 3.1 계층도

```
학습 트랙 (Track)
  └─ 코스 (Course) — 교재 모음
       └─ 교재 (Material) — 단일 학습 단위
            └─ 스테이지 (Stage) — 학습 활동 단위 (6종 플러그인)
```

### 3.2 각 계층 상세

#### 트랙 (Track)

| 속성 | 설명 |
|------|------|
| **정의** | 하나의 학습 목표를 달성하기 위한 코스·교재의 순서 있는 묶음 |
| **유형** | `industry` (산업별), `skill` (기술별), `custom` (커스텀) |
| **진행 모드** | 순차 강제, 순차 선택, 자유 |
| **연계** | Raid/해커톤과 연계하여 참가자 사전 학습 경로로 활용 |
| **AI 생성** | 트랙 전체 커리큘럼 AI 자동 설계 + 일괄 교재 생성 지원 |

#### 코스 (Course)

| 속성 | 설명 |
|------|------|
| **정의** | 교재(Material)의 논리적 그룹. 하나의 주제 또는 모듈 단위 |
| **진행 모드** | `sequential_forced`, `sequential_optional`, `free` |
| **상태** | `draft`, `published`, `archived` |
| **필수/선택** | 트랙 내에서 필수 또는 선택 코스로 지정 가능 |

#### 교재 (Material)

| 속성 | 설명 |
|------|------|
| **정의** | 학습의 기본 단위. 여러 스테이지로 구성 |
| **난이도** | `beginner`, `intermediate`, `advanced` |
| **상태** | `draft` → `published` → `archived` |
| **메타데이터** | 제목, 설명, 썸네일, 카테고리, 태그, 예상 소요시간, 학습 목표 |
| **AI 생성** | Gemini 기반 자동 생성 지원 |

#### 스테이지 (Stage)

| 속성 | 설명 |
|------|------|
| **정의** | 교재 내 개별 학습 활동. 화면 하나에 대응 |
| **유형** | 6종 플러그인 기반 (content, quiz, prompt, flashcard, pdf, branching-lesson) |
| **순서** | `orderIndex`로 정렬 |
| **슬러그** | URL-friendly 식별자 |
| **팁 콘텐츠** | 학습 중 도움말로 표시되는 부가 설명 |

## 4. Layer 1 — 스테이지 플러그인 시스템 (Phase 2 ✅)

### 4.1 플러그인의 의미

**한 문장 정의**: "스테이지 타입 하나를 추가하려면 파일 하나만 작성하면 되는 구조."

#### 도입 전 (10개 파일 수정 필요)

| # | 파일 | 수정 내용 |
|---|------|----------|
| 1 | `shared/schema/learning.ts` | `STAGE_TYPES` 배열 수정 |
| 2 | `LearningStageRenderer.tsx` | switch문에 case 추가 |
| 3 | 새 Viewer 컴포넌트 | 렌더링 구현 |
| 4 | 타입 정의 | 데이터 필드 추가 |
| 5 | DB 스키마 | jsonb 컬럼 추가 |
| 6 | AI 파이프라인 | 프롬프트 규칙 추가 |
| 7 | Wizard | 선택지 추가 |
| 8~10 | 검증/테스트/문서 | 각 단계별 |

#### 도입 후 (2파일 + 레지스트리 2줄)

```
1. server/learning/plugins/{type}.ts     — StagePlugin 구현 (1파일)
2. client/src/learning/plugins/{type}.ts  — ClientStagePlugin 어댑터 (1파일)
3. server/learning/plugins/registry.ts    — import + Map 등록 (1줄)
4. client/src/learning/plugins/client-registry.ts — import + Map 등록 (1줄)
5. shared/schema/learning.ts의 STAGE_TYPES — 타입 추가 (1줄)
```

### 4.2 서버-클라이언트 대칭 플러그인 구조

```
Server Plugin                          Client Plugin
┌──────────────────────┐               ┌──────────────────────┐
│ StagePlugin<T,R>     │               │ ClientStagePlugin    │
│ ──────────────────── │               │ ──────────────────── │
│ id: string           │               │ id: string           │
│ label: string        │               │ label: string        │
│ description: string  │               │ Viewer: Component    │
│ dataSchema: Zod      │     미러      │ showInWizard: bool   │
│ aiPromptFragment?    │ ←──────────→  │                      │
│ scorable: boolean    │               │                      │
│ toEvents()           │               │                      │
│ score?()             │               │                      │
└──────────────────────┘               └──────────────────────┘
```

### 4.3 등록된 6종 플러그인

| 플러그인 ID | 라벨 | 채점 | 목적 | AI 생성 |
|------------|------|------|------|---------|
| `content` | 콘텐츠 | ✗ | 마크다운 기반 텍스트 학습 | ✅ |
| `quiz` | 퀴즈 | ✅ | 4지선다형 객관식 (80% 통과) | ✅ |
| `prompt` | 프롬프트 | ✅ | AI 프롬프트 작성 실습 (유사도 0~100) | ✅ |
| `flashcard` | 플래시카드 | ✅ | 앞/뒤 카드 암기 학습 (Known/Unknown) | ✅ |
| `pdf` | PDF | ✗ | PDF 문서 뷰어 | ✗ |
| `branching-lesson` | 분기 학습 | ✗ | 시나리오 기반 트리 구조 학습 | ✅ |

### 4.4 각 플러그인 상세

#### 퀴즈 (Quiz)

```typescript
// 데이터 형식 (quizData jsonb)
{
  "question": "질문 텍스트",
  "options": [
    { "id": "opt_0", "text": "선택지 1" },
    { "id": "opt_1", "text": "선택지 2" },
    { "id": "opt_2", "text": "선택지 3" },
    { "id": "opt_3", "text": "선택지 4" }
  ],
  "correctAnswer": "opt_1",
  "explanations": { "opt_1": "정답 해설" },
  "hasExplanation": true
}
```
- 채점: `plugin.score()` → `{ points: 정답수, maxPoints: 총문제수, passed: 80%이상 }`
- 이벤트: `attempted` → `answered` → `passed`/`failed`
- 시도 기록: `learningQuizAttempts` 테이블

#### 프롬프트 실습 (Prompt)

```typescript
// 데이터 형식 (promptData jsonb)
{
  "instruction": "실습 과제 안내",
  "sampleAnswer": "모범 답안",
  "hasSampleAnswer": true
}
```
- 채점: AI 유사도 평가 (의도/구조/명확성 3축, 0~100점)
- 이벤트: `attempted` → `submitted`
- 시도 기록: `learningPromptAttempts` 테이블 (유사도, 점수 내역, AI 피드백)

#### 플래시카드 (Flashcard)

```typescript
// 데이터 형식 (flashcardData jsonb)
{
  "cards": [
    { "id": "card_0", "front": "앞면 질문", "back": "뒷면 답변", "difficulty": "medium" }
  ],
  "settings": { "shuffleOnLoad": true, "showProgress": true, "passingThreshold": 80 }
}
```
- 채점: Known/Unknown 비율
- 이벤트: `attempted` → `passed`/`failed`
- 시도 기록: `learningFlashcardAttempts` 테이블

## 5. Layer 2 — 학습 이벤트 스트림 (Phase 1 ✅)

### 5.1 핵심 개념

xAPI (Experience API)를 경량화한 **단일 이벤트 테이블** (`learning_events`). 모든 학습 상호작용은 `{ user, verb, objectType, objectId, result, context, occurredAt }` 형태로 기록된다.

### 5.2 이벤트 체계

#### 12종 Verb

| Verb | 설명 | 발생 시점 |
|------|------|----------|
| `viewed` | 콘텐츠 조회 | 교재/트랙 페이지 열람 |
| `started` | 학습 시작 | 교재/트랙/세션 시작 |
| `attempted` | 시도 | 퀴즈/프롬프트/플래시카드 시작 |
| `answered` | 응답 | 퀴즈 답 제출 |
| `completed` | 완료 | 교재/스테이지/트랙 완료 |
| `passed` | 통과 | 퀴즈 80% 이상 |
| `failed` | 미통과 | 퀴즈 80% 미만 |
| `submitted` | 제출 | 프롬프트 답변 제출 |
| `commented` | 코멘트 | 학습 노트 작성 |
| `branched` | 분기 | 분기 학습 선택 |
| `abandoned` | 이탈 | 학습 도중 이탈 |
| `resumed` | 재개 | 이탈 후 학습 재개 |

#### 5종 Object Type

`material`, `stage`, `question`, `assignment`, `track`

### 5.3 이벤트 수집 구조

```
[학습 라우트] → eventStream.record() / recordBatch() (fire-and-forget)
     │
     ▼
[learning_events 테이블] ← 3개 인덱스 (userId, objectType+objectId, occurredAt)
     │
     ├─ [eventAggregator] ← 1분 주기 배치 집계 → learningDailyActivity 업데이트
     │
     └─ [analyticsQueries] ← 7대 분석 지표 실시간 조회
```

#### 수집 포인트 (5개 라우트)

| 라우트 파일 | 수집 이벤트 |
|-----------|-----------|
| `public-attempts.ts` | 퀴즈/프롬프트/플래시카드 `attempted`/`answered`/`passed`/`failed` |
| `public-progress.ts` | 스테이지/교재/트랙 `completed` |
| `public-materials.ts` | 교재 상세 `viewed` (인증 사용자만) |
| `session-logs.ts` | 세션 `started`/`abandoned` |
| `tracks.ts` | 트랙 상세 `viewed` |

> **Phase 2 이후**: 플러그인의 `toEvents()` 함수가 이벤트를 자동 생성하므로, 수동 `eventStream.record()` 호출이 불필요해졌다.

### 5.4 배치 집계기 (Event Aggregator)

- 1분 주기 `setInterval` 스케줄러
- 슬라이딩 윈도우 방식으로 `learning_events` → `learningDailyActivity` 집계
- `sessionsCount` 업데이트 (Phase 2에서 일원화)
- 서버 시작/종료 시 자동 등록/해제

### 5.5 7대 분석 지표

| 지표 | 설명 |
|------|------|
| **활성 사용자** | DAU/WAU/MAU + 일별 시계열 |
| **교재 완료 퍼널** | 교재별 시작→완료 전환율 |
| **이탈 핫스팟** | 이탈률 높은 스테이지 Top 10 |
| **퀴즈 정확도 분포** | 5구간 히스토그램 (0-20/20-40/.../80-100) |
| **평균 세션 길이** | 평균/중앙값 (분) |
| **재시도 스테이지** | 가장 많이 재시도된 스테이지 Top 10 |
| **일별 이벤트 볼륨** | 일별 이벤트 수 추이 |

## 6. Layer 3 — Assessment: Gradebook 통합 성적부 (Phase 3-A 🔄)

### 6.1 문제 정의

**"학습자 A의 전체 성적을 한 화면에서 보는 방법이 없다."**

- 퀴즈/프롬프트/플래시카드 시도는 각각 테이블에 분산
- 트랙/교재 단위로 통합된 성적 뷰 없음
- 관리자가 학습자별 진도/성적을 한눈에 파악 불가

### 6.2 Gradebook 서비스 설계

**신규 DB 테이블 없이** 기존 attempts 3종 + 진도 테이블을 Gradebook 서비스가 조합한다.

```typescript
interface GradebookEntry {
  stageId: string;
  stageTitle: string;
  stageType: string;          // 'quiz' | 'prompt' | 'flashcard' | 'content' | 'pdf'
  scorable: boolean;
  bestScore: number | null;    // 시도 중 최고 점수
  maxScore: number | null;     // 만점
  passed: boolean;             // 통과 여부 (plugin.score() 기준)
  attempts: number;            // 총 시도 횟수
  lastAttemptAt: Date | null;
  completed: boolean;          // 스테이지 완료 여부
}

interface MaterialGradebook {
  materialId: string;
  materialTitle: string;
  totalStages: number;
  completedStages: number;
  scorableStages: number;
  averageScore: number | null;  // scorable 스테이지 평균 점수율 (%)
  entries: GradebookEntry[];
}
```

### 6.3 채점 로직 (plugin.score() 재활용)

| 스테이지 유형 | 점수 소스 | 만점 기준 | 통과 기준 |
|-------------|----------|----------|----------|
| `quiz` | `MAX(score)` from learningQuizAttempts | 총 문제 수 | 80% 이상 |
| `prompt` | `MAX(similarityScore)` from learningPromptAttempts | 100 (유사도 만점) | plugin.score() 판정 |
| `flashcard` | `MAX(knownCount)` from learningFlashcardAttempts | 총 카드 수 | passingThreshold 이상 |
| `content` | scorable: false | — | 완료 여부만 |
| `pdf` | scorable: false | — | 완료 여부만 |

## 7. AI 교재 생성 파이프라인

### 7.1 개요

DAKER LMS의 핵심 차별 기능으로, **AI가 교재를 자동 생성**한다. Gemini와 GPT의 **상호 검수(Mutual Review)** 4단계 파이프라인을 통해 품질을 보장한다.

### 7.2 파이프라인 구조

```
[관리자: 주제/키워드 입력]
            │
            ▼
┌─────────────────────────────────┐
│  Step 1: 커리큘럼 생성 (Gemini) │
│  주제 분석 → 학습 목표 도출     │
│  → 스테이지 구성 계획           │
└──────────────┬──────────────────┘
               │
               ▼
┌─────────────────────────────────┐
│  Step 2: 상호 검수 (GPT)       │
│  Gemini 결과를 GPT가 리뷰      │
│  → 개선점 피드백               │
└──────────────┬──────────────────┘
               │
               ▼
┌─────────────────────────────────┐
│  Step 3: 콘텐츠 생성 (Gemini)  │
│  각 스테이지별 콘텐츠 생성      │
│  → 퀴즈, 프롬프트, 플래시카드  │
└──────────────┬──────────────────┘
               │
               ▼
┌─────────────────────────────────┐
│  Step 4: 최종 검수 & 확정      │
│  관리자 미리보기 → 수정 → 확정  │
└─────────────────────────────────┘
```

### 7.3 플러그인 연동 (Phase 2 이후)

각 서버 플러그인의 `aiPromptFragment` 속성에 해당 스테이지 유형의 데이터 형식이 한국어로 명시되어 있으며, AI 교재 파이프라인이 이를 **자동으로** 프롬프트에 포함한다.

- `quiz` 플러그인 → 퀴즈 데이터 형식 규칙이 자동 주입
- `flashcard` 플러그인 → 플래시카드 형식 규칙이 자동 주입
- `pdf` 플러그인 → AI 생성 대상 아님 (자동 제외)
- **새 플러그인 추가 시** `aiPromptFragment`만 작성하면 AI 파이프라인이 자동 인식

### 7.4 AI 생성 기능 목록

| 기능 | 설명 | 스트리밍 |
|------|------|---------|
| **단일 교재 생성** | 주제/키워드로 전체 교재 생성 | SSE |
| **일괄 교재 생성** | 트랙 내 여러 교재를 한 번에 생성 | SSE |
| **커리큘럼 AI 생성** | 트랙 전체 커리큘럼 자동 설계 | SSE |
| **커리큘럼 AI 보완** | 기존 커리큘럼 개선 제안 | SSE |
| **프롬프트 스펙 파싱** | 자연어 명세서로부터 교재 구조 생성 | SSE |
| **학습 AI 어시스턴트** | 학습 중 질의응답 챗봇 | SSE |

### 7.5 일일 토큰 제한

AI 생성에는 일일 토큰 한도가 적용되며, `aiGenerationLogs` 테이블에 입력/출력 토큰, 사용 여부, 피드백이 기록된다.

## 15. 학습 그룹 및 일괄 등록 시스템

### 15.1 학습 그룹 (Learning Groups)

학습 그룹은 **수강생 묶음 + 권한 관리 + 출결 관리**의 중심 엔티티이다. 기업 교육, 대학 수업 등 조직 단위 학습 운영을 위한 핵심 인프라.

#### 그룹 구조

```
학습 그룹 (Learning Group)
  ├─ 그룹 멤버 (N명) — 이메일 기반 등록
  ├─ 교재/트랙 권한 — 기간 조건부 수강 권한
  ├─ 출결 기록 — 자동 감지 + 수동 편집
  └─ 출결 설정 — 최소 인정 시간 등 임계값
```

#### 데이터 모델

**`learning_groups`** — 그룹 컨테이너

| 컬럼 | 타입 | 설명 |
|------|------|------|
| `id` | varchar (PK) | 그룹 ID |
| `name` | varchar(200) | 그룹명 |
| `description` | text | 그룹 설명 |
| `createdBy` | varchar (FK → users) | 생성자 |
| `createdAt`, `updatedAt` | timestamp | 생성/수정 시각 |

**`learning_group_members`** — 그룹 멤버십

| 컬럼 | 타입 | 설명 |
|------|------|------|
| `id` | varchar (PK) | 멤버십 ID |
| `groupId` | varchar (FK → learning_groups, CASCADE) | 소속 그룹 |
| `userId` | varchar (FK → users, CASCADE) | 사용자 |
| `memo` | text | 멤버 메모 |
| `addedBy` | varchar | 등록자 |
| `addedAt` | timestamp | 등록 시각 |
| **UK** | (groupId, userId) | 중복 방지 |

#### 그룹 관리 API

| Method | Endpoint | 설명 |
|--------|----------|------|
| GET | `/api/learning/admin/groups` | 그룹 목록 (검색, 페이징) |
| POST | `/api/learning/admin/groups` | 그룹 생성 |

- **그룹 목록**: `/admin/learning` > "그룹" 탭 — 검색, 페이징

  - 출석 관리: 그리드 UI (→ 16장)

### 15.2 일괄 등록 (Bulk Enrollment)

최대 **1,000명**까지 한 번에 수강 등록하는 4단계 마법사. `one-shot` (즉석 등록) 또는 `save-group` (그룹 저장 후 등록) 모드 지원.

#### 4단계 마법사

```
Step 1: 멤버 입력 — 이메일 붙여넣기 / CSV 업로드
           │
           ▼
Step 2: 교재/트랙 선택 — 체크박스로 수강 대상 지정
           │
           ▼
Step 3: 기간 설정 — 시작일/종료일 (선택)
           │
           ▼
Step 4: 리뷰 & 실행 — dryRun 미리보기 → 확정 실행
```

#### 처리 파이프라인 (3-Phase)

| Phase | 처리 | 설명 |
|-------|------|------|
| **1. Resolve** | 이메일 → userId 매핑 | 정규화, 중복 제거, 미등록자 건너뜀 |
| **2. Assign** | (user × target) 멱등 처리 | 활성 기존 → `skipped`, 취소 → `restored`, 신규 → `created` |
| **3. Execute** | save-group 시 그룹 생성 | 그룹 + 멤버 + 권한 원자적 처리 |

#### 진도 테이블 확장 (기존 테이블에 컬럼 추가)

`user_course_progress`, `user_track_progress`에 추가된 컬럼:

| 컬럼 | 타입 | 설명 |
|------|------|------|
| `enrolledBy` | varchar (FK → users) | 등록 처리한 관리자 |
| `enrolledAt` | timestamp | 등록 시각 |
| `enrollmentSource` | enum | `admin_assigned`, `group`, `payment`, `self_enrolled` |
| `startsAt` | timestamp | 수강 시작일 (선택) |
| `expiresAt` | timestamp | 수강 종료일 (선택) |
| `sourceGroupId` | varchar (FK → learning_groups) | 권한 부여 그룹 |

#### 일괄 등록 API

| Method | Endpoint | 설명 |
|--------|----------|------|
| POST | `/api/learning/admin/bulk-enroll` | 일괄 등록 (dryRun 지원, 최대 1,000명) |

**요청**:
```typescript
{
  mode: 'one-shot' | 'save-group',
  group?: { name, description },
  members: [{ email, name?, memo? }],  // 1~1,000
  targets: { materialIds: string[], trackIds: string[] },
  period?: { startsAt?, expiresAt? },
  dryRun: boolean
}
```

**응답**: 멤버별 처리 결과 + 과제별 처리 결과 + 요약 통계

## 16. 경량 출결 시스템

### 16.1 설계 철학

**"세션 로그는 이미 쌓이고 있다 — 별도 하드웨어나 복잡한 QR 시스템 없이, 기존 학습 세션 로그를 재활용하여 출결을 자동 감지한다."**

- 기존 `learning_session_logs` 테이블의 학습 시간을 집계
- 임계값(`minDurationSeconds`, 기본 60초) 이상이면 출석 인정
- 관리자는 그리드 UI에서 원클릭으로 수동 편집 가능

### 16.2 출결 상태

| 상태 | 라벨 | 설명 |
|------|------|------|
| `present` | 출석 | 최소 인정 시간 이상 학습 |
| `absent` | 결석 | 학습 기록 없음 또는 미달 |
| `late` | 지각 | 관리자 수동 지정 |
| `excused` | 공결 | 관리자 수동 지정 |
| `early_leave` | 조퇴 | 관리자 수동 지정 |

### 16.3 데이터 모델

**`learning_attendance_records`** — 출결 기록

| 컬럼 | 타입 | 설명 |
|------|------|------|
| `id` | varchar (PK) | 기록 ID |
| `groupId` | varchar (FK → learning_groups, CASCADE) | 소속 그룹 |
| `userId` | varchar (FK → users, CASCADE) | 대상 학습자 |
| `date` | date | 출결 날짜 (KST 기준 YYYY-MM-DD) |
| `status` | enum | present / absent / late / excused / early_leave |
| `checkInAt`, `checkOutAt` | timestamp | 첫 세션 시작 / 마지막 세션 종료 |
| `durationSeconds` | integer | 해당일 총 학습 시간 (초) |
| `detectionSource` | enum | `auto` (자동 감지) / `manual` (수동 편집) |
| `memo` | text | 사유 메모 |
| `recordedBy` | varchar | 마지막 수정 관리자 |
| **UK** | (groupId, userId, date) | 1인 1일 1건 보장 |

**`learning_attendance_settings`** — 그룹별 출결 설정

| 컬럼 | 타입 | 설명 |
|------|------|------|
| `id` | varchar (PK) | 설정 ID |
| `groupId` | varchar (UK, FK → learning_groups, CASCADE) | 대상 그룹 |
| `minDurationSeconds` | integer (default: 60) | 출석 인정 최소 학습 시간 (초) |
| `updatedBy` | varchar | 마지막 수정자 |

### 16.4 자동 감지 로직

```
[학습 세션 로그 조회]
  그룹 멤버 ∩ 그룹 교재 ⊂ learning_session_logs
       │
       ▼
[일별 집계 (KST 기준)]
  SUM(durationSeconds) per (userId, date)
       │
       ▼
[임계값 판정]
  durationSeconds >= minDurationSeconds → 'present'
  durationSeconds < minDurationSeconds  → 'absent'
       │
       ▼
[그리드 셀 반환]
  persisted: false (아직 DB 미저장)
  detectionSource: 'auto'
```

- **스냅샷 저장**: 관리자가 "스냅샷 저장" 실행 시 자동 감지 결과를 DB에 일괄 저장 (기존 수동 편집은 보존)
- **수동 편집**: 관리자가 셀 클릭으로 상태 변경 시 `detectionSource='manual'`로 기록 (자동 감지에 의해 덮어씌워지지 않음)

### B. 디자인 시스템

DAKER의 **Editorial Architect** 디자인 시스템을 따른다.

- **3-Color 원칙**: Blue (#0053db) + Slate (#2a3439) + White (#ffffff)
- **3 모드**: Light / Dark / Brutal
- **No-Line 규칙**: 1px 보더 대신 배경색 톤 전환으로 영역 구분
- **카드 그리드**: `grid-cols-1 min-[600px]:grid-cols-2 xl:grid-cols-3 gap-4 min-[600px]:gap-6`

### C. 접근성

- WCAG AA 준수 (텍스트 대비율 4.5:1 이상)
- 키보드 내비게이션 완벽 지원
- ARIA 라벨 모든 인터랙티브 요소 적용
- 터치 타겟 최소 44px × 44px

### D. 한국어 최적화

- 모든 UI 텍스트, 라벨, 에러 메시지 한국어
- 날짜 형식: `YYYY년 MM월 DD일 HH:mm`
- 숫자 형식: 천 단위 구분자 (`1,000,000`)
- 서체: Plus Jakarta Sans + Pretendard
