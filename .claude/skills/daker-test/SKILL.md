# DAKER Test — 종합 테스트 + E2E 검증 + 리포트 자동 생성

코드 변경 후 "잘 돌아가겠지"라는 가정은 위험하다. 이 스킬은 단위 테스트, E2E 브라우저 검증, 콘솔/네트워크 에러 수집을 병렬로 수행하고, 결과를 하나의 테스트 리포트로 통합한다.

## 전체 흐름

```
Phase 1: 사전 준비 (서버 상태, 인증 세션 확인)
  ↓
Phase 2: 병렬 테스트 실행
  ├─ 에이전트 A: 단위 테스트 (Vitest)
  ├─ 에이전트 B: E2E 페이지 검증 (Playwright MCP)
  └─ 에이전트 C: 회귀 테스트 (변경 코드 영향 분석)
  ↓
Phase 3: 결과 수집 & 리포트 생성
```

---

## Phase 1: 사전 준비

### 1-1. 개발 서버 확인

```bash
curl -s -o /dev/null -w "%{http_code}" http://localhost:5000
```

- 응답 코드 200이면 서버 정상
- 아니면 `npm run dev` 실행 안내

### 1-2. SSO 인증 세션 (E2E에 필요할 때만)

인증이 필요한 페이지(admin, expert, 마이페이지 등)를 테스트하려면:

1. `e2e/.auth/user.json` 파일 존재 여부 확인
2. 없으면 사용자에게 안내:
   ```
   SSO 로그인 세션이 필요합니다.
   `npx playwright test e2e/save-auth.ts --headed` 를 실행하고
   브라우저에서 DACON 로그인을 완료해주세요.
   ```
3. 인증 없이 테스트 가능한 페이지(public)는 그대로 진행

### 1-3. 테스트 범위 결정

사용자 요청에 따라 범위를 결정한다:

| 요청 | 범위 |
|------|------|
| "전체 테스트" | 단위 + E2E 전 페이지 + 회귀 |
| "E2E만" | Playwright 페이지 검증만 |
| "스모크 테스트" | 주요 6개 페이지만 빠르게 |
| "회귀 테스트" | git diff 기반 변경 영향 분석 |
| "PR 전 검증" | 단위 + 변경 관련 E2E + 회귀 |

## Phase 2: 병렬 테스트 실행

독립적인 테스트는 반드시 병렬 에이전트로 동시에 실행한다. 단일 메시지에서 여러 Agent 호출을 보내면 된다.

### 에이전트 A: 단위 테스트

```
DAKER 프로젝트의 단위 테스트를 실행하고 결과를 분석해줘.

실행:
1. `npm test` — 전체 Vitest 테스트 실행
2. 결과에서 다음을 추출:
   - 총 테스트 파일 수, 통과/실패/스킵 수
   - 총 테스트 케이스 수, 통과/실패/스킵 수
   - 실행 시간
3. 실패한 테스트가 있으면:
   - 파일명, 테스트명, 에러 메시지 기록
   - 실패 원인 간단 분석

결과를 다음 형식으로 보고:
- 전체: X파일, Y테스트
- 통과: X / 실패: Y / 스킵: Z
- 실행 시간: Xs
- [실패 상세 있으면 포함]
```

### 에이전트 B: E2E 페이지 검증 (Playwright MCP)

Playwright MCP 도구를 사용하여 실제 브라우저에서 페이지별 검증을 수행한다.

**검증 대상 페이지 (스모크 테스트 기본셋):**

| # | 페이지 | URL | 인증 필요 |
|---|--------|-----|-----------|
| 1 | 메인 | `http://localhost:5000/` | ❌ |
| 2 | 해커톤 목록 | `http://localhost:5000/public/hackathons` | ❌ |
| 3 | Raids 목록 | `http://localhost:5000/public/raids` | ❌ |
| 4 | 베이스캠프 | `http://localhost:5000/public/basecamp` | ❌ |
| 5 | 커뮤니티 | `http://localhost:5000/community` | ❌ |
| 6 | 학습 | `http://localhost:5000/public/learning` | ❌ |

**페이지별 검증 절차:**

```
각 페이지에 대해:
1. browser_navigate → 페이지 이동
2. browser_snapshot → DOM 구조 확인 (렌더링 성공 여부)
3. browser_console_messages → 콘솔 에러 수집
4. browser_network_requests → 실패한 API 호출 수집
5. browser_take_screenshot → 스크린샷 캡처
   - 저장 경로: e2e/screenshots/report-YYYY-MM-DD/
   - 파일명: XX-페이지명.png
```

**콘솔 에러 필터링 규칙:**
- Google Analytics/Attestation 에러 → 무시 (외부 서비스)
- 401 Unauthorized (비로그인 상태) → 예상된 동작으로 기록
- 그 외 에러 → 주의 필요로 분류

**네트워크 요청 필터링:**
- google-analytics, googletagmanager → 무시
- localhost API 4xx/5xx → 기록

### 에이전트 C: 회귀 테스트

```
git diff로 변경된 파일을 분석하고 회귀 위험을 평가해줘.

실행:
1. `git diff --name-only HEAD~5` (또는 dev 브랜치 기준)으로 변경 파일 목록 확보
2. 변경된 파일 각각에 대해:
   - 관련 테스트 파일이 있는지 확인
   - 해당 테스트가 통과하는지 확인
   - 다른 파일에서 이 파일을 import하는 곳 파악 (영향 범위)
3. 영향 범위 평가:
   - 서버 API 변경 → 해당 API를 호출하는 클라이언트 코드 확인
   - 스키마 변경 → 관련 라우트, 서비스 확인
   - 공통 컴포넌트 변경 → 사용처 전수 조사

결과를 다음 형식으로 보고:
- 변경 파일: X개
- 영향 범위: [파일 목록]
- 테스트 커버리지: X/Y 파일에 테스트 있음
- 회귀 위험: HIGH/MEDIUM/LOW
- [위험 항목 상세]
```

## Phase 3: 결과 수집 & 리포트 생성

### 3-1. 리포트 파일

**파일 경로**: `docs/plans/YYYY-MM-DD-종합테스트_완료리포트.md`

`references/test-report-template.md`를 참고하여 작성한다.

### 3-2. 리포트 구성

```markdown
# 종합 테스트 완료 리포트

## 테스트 개요
- 실행 일시: YYYY-MM-DD HH:mm
- 테스트 범위: [전체/스모크/회귀/PR검증]
- 브랜치: [현재 브랜치명]

## 1. 단위 테스트 결과
| 항목 | 수치 |
|------|------|
| 테스트 파일 | X개 (통과: Y, 실패: Z, 스킵: W) |
| 테스트 케이스 | X개 (통과: Y, 실패: Z, 스킵: W) |
| 실행 시간 | X.XXs |

## 2. E2E 페이지 검증 결과
| # | 페이지 | 렌더링 | 콘솔 에러 | API 에러 | 스크린샷 |
|---|--------|--------|-----------|----------|---------|
| 1 | 메인 | ✅ | 0 | 0 | 파일명 |

## 3. 콘솔 에러 분석
### 무시 가능 (외부 서비스)
- Google Analytics Attestation: X건

### 예상된 에러 (비로그인)
- 401 Unauthorized: X건

### 주의 필요
- [있으면 상세 기술]

## 4. 회귀 테스트 결과
- 변경 파일: X개
- 영향 범위: [파일 목록 또는 요약]
- 회귀 위험도: LOW/MEDIUM/HIGH

## 5. 종합 판정
- 단위 테스트: ✅ ALL PASS / ❌ X건 실패
- E2E 검증: ✅ ALL PASS / ⚠️ X건 주의
- 회귀 위험: LOW / MEDIUM / HIGH
- **최종 판정**: ✅ 배포 가능 / ⚠️ 조건부 가능 / ❌ 수정 필요
```

### 3-3. 스크린샷 정리

```
e2e/screenshots/report-YYYY-MM-DD/
├── 01-main-page.png
├── 01-main-page-console.log
├── 01-main-page-network.log
├── 02-hackathons.png
├── 02-hackathons-console.log
...
```

각 페이지마다 스크린샷(.png)과 콘솔 로그(.log), 네트워크 로그(.log)를 함께 저장한다.

## 특수 시나리오

### PR 전 검증 모드

1. `git diff dev...HEAD --name-only`로 변경 파일 목록 확보
2. 변경 파일과 관련된 테스트만 선별 실행
3. 변경 파일과 관련된 페이지만 E2E 검증
4. 리포트에 "PR 검증" 범위 명시

### 특정 기능 테스트

사용자가 특정 기능(예: "해커톤 참가 신청 테스트해줘")을 요청하면:

1. 해당 기능의 API 엔드포인트 파악
2. 관련 단위 테스트 선별 실행: `npx vitest run [관련파일]`
3. Playwright로 해당 기능 시나리오 검증
4. 축소된 리포트 생성

### 타입 체크 포함

전체 테스트에는 타입 체크도 포함한다:

```bash
npx tsc --noEmit
```

에러가 있으면 리포트 "추가 검증" 섹션에 기록한다.

## 핵심 원칙

1. **병렬 먼저**: 독립적인 테스트는 항상 병렬 에이전트로 실행하여 시간을 단축한다.
2. **증거 남기기**: 스크린샷과 로그는 "통과했다"는 주장보다 강력한 증거다.
3. **노이즈 필터링**: Google Analytics 에러 같은 외부 서비스 노이즈는 분리하여 실제 문제에 집중한다.
4. **리포트는 판단을 포함해야 한다**: 숫자만 나열하지 말고 "배포 가능/불가" 최종 판정을 내린다.
